The problem

You stopped writing. But you still need to know what shipped.

Prompting replaced writing. Agents produce most of the code, most of the copy, most of the config. You review a diff, maybe. You approve a PR, maybe. But you do not fully understand what your AI built - and the documentation layer to surface it was never designed for this world.

Blackbox

AI ships code you did not write

Thousands of lines per week, across files you never opened. You are the accountable party for software you have never read line by line. The diff view is not enough.

Markdown

A format built for writers

Markdown exists to help humans author text efficiently. Humans are no longer the authors. Keeping a markdown pipeline for docs you no longer write is legacy infrastructure.

No memory

Every prompt starts from zero

Your agent has no record of what was approved last time, what patterns you accepted, what decisions you already made. It rediscovers your codebase on every task - and contradicts itself.

No audit

No record of what the AI changed

Commits tell you what files changed. They do not tell you what your AI decided, what it assumed, or what corners of the system it touched that you did not review.

The loop

Agents write. You approve. Agents obey.

Glassdocs establishes a three-step cycle between you and your AI. Each role has its own verb. Each step reinforces the next. Together they turn AI-authored code from a blackbox into a system you actually govern.

1

Agents write the docs

Every change your AI makes comes with documentation - written by the agent, into docs/, as HTML. One commit carries both the code and the human-readable explanation of what happened.

2

You read and approve

You do not write docs. You read them, curate them, and accept them as the record of truth. Visual, navigable, renders in any browser. Edit what is wrong, approve what is right.

3

Agents obey the docs

On every future prompt, the agent reads the approved docs before touching code. Work that contradicts them is flagged and rejected. The docs become the contract - enforced, not aspirational.

How it works

Drop-in for any repo. No build step. No framework.

  1. 1

    Add Glassdocs to your repo

    One command scaffolds a docs/ folder, the navigation shell, and the agent instructions that tell every future agent how to write into it.

  2. 2

    Your agent works. It also documents.

    Whatever your agent builds - features, refactors, migrations - lands with an HTML doc page explaining what, why, and where. No separate documentation task. No drift.

  3. 3

    You review the docs in the PR

    The docs ARE the diff you read. Visual, structured, navigable. You approve or edit the docs - and the code lands alongside.

  4. 4

    Future agents read before they write

    The Glassdocs extension feeds approved docs back into every new agent session. Your AI cannot contradict what you already signed off on.
